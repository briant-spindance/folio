OAuth Integration

forge/features/oauth-integration/FEATURE.md  ·  Last modified Apr 9, 2026

Summary

Integrate OAuth 2.0 for third-party login providers, allowing users to authenticate via Google, GitHub, and Microsoft accounts instead of (or in addition to) email/password credentials.

This feature builds on top of the existing User Authentication system and adds federated identity support without replacing the current email/password flow.

Motivation

User research shows that 62% of users prefer social login over creating a new username/password combination. Supporting OAuth providers will:

  • Reduce friction during signup (fewer form fields, no password to remember)
  • Improve security posture (delegating auth to providers with 2FA infrastructure)
  • Enable future integrations (e.g., importing GitHub repos, Google Drive docs)
  • Align with enterprise SSO requirements for future B2B sales

Acceptance Criteria

  • User can sign up / log in with Google OAuth
  • User can sign up / log in with GitHub OAuth
  • User can sign up / log in with Microsoft OAuth
  • Existing users can link/unlink OAuth providers in account settings
  • OAuth tokens are stored securely and refreshed automatically
  • Login page shows provider buttons with proper branding
  • Error handling for denied permissions, expired tokens, provider outages

Technical Approach

Provider Configuration

Each OAuth provider will be configured via environment variables following the pattern:

OAUTH_GOOGLE_CLIENT_ID=...
OAUTH_GOOGLE_CLIENT_SECRET=...
OAUTH_GOOGLE_REDIRECT_URI=https://app.example.com/auth/callback/google

OAUTH_GITHUB_CLIENT_ID=...
OAUTH_GITHUB_CLIENT_SECRET=...
OAUTH_GITHUB_REDIRECT_URI=https://app.example.com/auth/callback/github

A provider is automatically enabled when its CLIENT_ID and CLIENT_SECRET are set. The login page dynamically renders buttons for enabled providers only.

Auth Flow

  1. User clicks "Sign in with Google" on the login page
  2. Server generates a state token (CSRF protection) and redirects to the provider's authorization endpoint
  3. User authenticates with the provider and grants permissions
  4. Provider redirects back to /auth/callback/:provider with an authorization code
  5. Server exchanges the code for an access token and fetches the user profile
  6. Server matches the profile to an existing user (by email) or creates a new account
  7. Server issues a session token and redirects to the dashboard

Data Model Changes

A new oauth_identities table links OAuth provider accounts to local user records:

Column Type Description
id uuid Primary key
user_id uuid FK to users table
provider string "google", "github", "microsoft"
provider_user_id string Unique ID from the provider
access_token encrypted string Current access token
refresh_token encrypted string Refresh token (if available)
token_expires_at timestamp Access token expiry
created_at timestamp When the link was created

Security Considerations

  • State parameter — CSRF protection using a server-generated, short-lived token stored in the session
  • Token storage — Access and refresh tokens encrypted at rest using AES-256-GCM
  • Scope minimization — Request only openid, email, and profile scopes
  • Email verification — Only accept verified email addresses from providers
  • Account linking — Require the user to be logged in before linking a new provider (prevents account takeover)

Open Questions

Should we support SAML/OIDC for enterprise SSO in this feature, or defer that to a separate feature?

Do we need to handle the case where a user signs up with Google and later tries to create an account with the same email via email/password? Current proposal: merge the accounts automatically if the email is verified by both sources.
