Project Brief

forge/wiki/project-brief.md  ·  Last modified Apr 1, 2026

Overview

Forge is a local-first, file-based project management tool designed for software development teams that want their project state to live alongside their code. Instead of a hosted SaaS platform, Forge stores everything — features, issues, sprints, docs, and configuration — as structured Markdown and YAML files in a forge/ directory within the project repository.

This approach gives teams full version control over their project management artifacts, enables AI agents to read and modify project state directly, and eliminates vendor lock-in.

Guiding Principles

  1. Files over databases. All project state is stored as human-readable Markdown and YAML files in the repository. No external database, no proprietary format.
  2. CLI-first, UI-enhanced. Every operation is available via the forge CLI. The web UI is a companion that reads and writes the same files.
  3. AI-native. The file-based structure is designed to be readable and writable by AI agents. The built-in chat interface provides context-aware assistance.
  4. Convention over configuration. Forge works out of the box with sensible defaults. Configuration is optional and progressive.
  5. Git-friendly. All operations produce clean diffs. Merge conflicts in project management files are rare and easy to resolve.

Project Structure

Forge creates and manages a forge/ directory in the project root:

forge/
├── forge.yaml              # Project configuration
├── wiki/                   # Project documentation
│   ├── project-brief.md
│   └── architecture.md
├── features/
│   ├── backlog.md          # Ordered feature backlog
│   ├── user-authentication.md
│   └── oauth-integration.md
├── issues/
│   ├── login-timeout.md
│   └── upload-progress-bar.md
└── sprints/
    ├── sprint-1.md
    ├── sprint-2.md
    └── sprint-3.md

Entity Model

Features

Features are the primary planning unit. Each feature is a Markdown file with YAML frontmatter containing metadata (status, assignee, points, labels) and a body containing the feature description, acceptance criteria, and technical notes.

Field Type Required Description
status enum Yes Workflow state: draft, ready, in-progress, review, done
assignee string No Team member responsible
points integer No Story points estimate
labels string[] No Categorization tags

Issues

Issues track bugs, tasks, and enhancement requests. They follow the same file format as features but with a simpler status model (open / closed) and can be linked to a parent feature.

Sprints

Sprints are time-boxed iterations. Each sprint file contains metadata (dates, status, capacity, goal) and a list of committed features and issues with their point values.

Development Phases

Phase 1: Foundation

Build the CLI with core CRUD operations for features, issues, sprints, and project docs. Establish the file format conventions and implement the forge init scaffolding command.

Phase 2: Web UI

Build the React frontend with all page views (dashboard, docs, features, sprints, issues, reviews, configuration). Implement the Go HTTP server that serves the UI and exposes the REST API.

Phase 3: AI Integration

Add the chat panel with context-aware AI assistance. The AI can read project state and suggest changes. Integrate with multiple LLM providers (OpenAI, Anthropic, Google).

The goal is not to replace human judgment but to give teams a tool that understands their project context and can help with routine planning tasks.
